[Template: Cloud Credential Compromise Investigation Report]

INCIDENT TYPE: Cloud Credential Theft / IAM Abuse / Cloud Account Takeover

FOCUS AREAS:
- Identify which cloud credentials were compromised: IAM user keys, role credentials, OAuth tokens, service principal secrets
- Determine how credentials were obtained: phishing, exposed in code/CI pipeline, SSRF, instance metadata service abuse, leaked .env file
- Map all API calls made with the stolen credentials (CloudTrail, Azure Monitor, GCP Audit Logs)
- Assess what cloud resources the attacker enumerated, modified, or exfiltrated
- Look for persistence mechanisms: new IAM users, roles, access keys, backdoored Lambda/functions, cross-account trust modifications
- Determine if attacker attempted privilege escalation within the cloud environment
- Check for resource abuse: crypto mining, outbound data transfer, new EC2/VM instances spun up

KEY IOC TYPES:
- Compromised access key ID or client ID
- Source IPs used for API calls (look for VPN/TOR/cloud-provider IPs — attackers often use EC2 in another account)
- Unusual API call patterns (DescribeInstances, ListBuckets, GetSecretValue, AssumeRole)
- New IAM entities created (users, roles, policies)
- S3 buckets accessed or made public
- Secrets Manager or Parameter Store secrets accessed
- Regions not normally used by the organisation

MITRE TECHNIQUES TO CONSIDER:
- T1078.004 Valid Accounts: Cloud Accounts — using stolen cloud credentials
- T1552.005 Cloud Instance Metadata API — SSRF to instance metadata for credential theft
- T1552.001 Credentials In Files — hardcoded keys in repos, .env files, CI/CD logs
- T1098 Account Manipulation — creating new IAM backdoor users or access keys
- T1136.003 Create Cloud Account — attacker-controlled IAM user for persistence
- T1537 Transfer Data to Cloud Account — exfiltrating data to attacker-controlled bucket
- T1580 Cloud Infrastructure Discovery — enumerate running services and storage
- T1530 Data from Cloud Storage — direct S3/Blob/GCS access for data theft

REPORT GUIDANCE:
Cloud credential investigations move fast — API access keys can be used globally within seconds
of theft. Time-to-detection is critical. Establish the first malicious API call timestamp precisely
and compare it to when credentials were exposed (commit timestamp, phishing email, etc.).

CloudTrail and equivalent logs are the primary evidence source. Verify log completeness: was
CloudTrail enabled in all regions? Were management events logged? Data events (S3 object-level)
are often not enabled by default and may leave gaps.

Privilege escalation in cloud environments is subtle. Look for AssumeRole chains, PassRole abuse,
and policy attachment to existing identities. An attacker who can assume a high-privilege role
may not create any new users — all actions appear under legitimate role names.

REQUIRED SECTIONS:
- Credential type and exposure vector
- Timeline of API activity (first call to detection)
- Resources accessed or modified (S3 buckets, secrets, compute)
- Privilege escalation path (if any)
- Persistence mechanisms created (new keys, roles, backdoor functions)
- Blast radius: data exposure and resource abuse assessment
- Credential rotation and remediation scope
